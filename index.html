# NH Demand Bin Verification — Setup & Daily Use Your Firebase project (`demand-bin-verification`) is wired in. Three one-time steps, then it runs. ## One-time setup (~10 min) 1. **Enable Firestore** — Firebase console → Build → Firestore Database → Create (Production mode). 2. **Enable Anonymous sign-in** — Authentication → Sign-in method → Anonymous → Enable. (Lets the app reach the DB; the PIN screen is your actual access control.) 3. **Publish the rules** — Firestore → Rules → paste `firestore.rules` → Publish. (Replaces the temporary open rule that expires.) 4. **Host the file** so the team opens the same URL (Firebase Hosting, internal server, or shared drive). They all talk to the same database. ## First login & team setup - Open the app → enter the master PIN **1988** (this always works so you can't lock yourself out) → you're in as IC Manager. - Go to **Team & Settings** → **Add person** for each team member: name, their own PIN, role (Checker/Manager), Active. → **Save settings & team**. - From then on, everyone logs in with their own PIN. Deactivate (uncheck Active) instead of deleting to keep someone's history. Settings also holds your **Divisions** (each with its Apprise prefix, name, and its own missing/not-available bins) and the **re-check window**. The seven divisions are pre-loaded: GT/TAG, NH/NAPA, VP/VIP, FS/Foreside, EE/Ever Ellis, BC/Beachcombers, TP/Transpac — edit any division's special bins to match the exact bin names in your system. ## Daily flow **Morning — IC Manager:** 1. **Build & Assign** → pick the **Division** you're processing, then drop in that division's bin report + demand CSV. 2. Pick the **Demand window** (7/14/30/60/70/80/90) and confirm the **Demand column** (auto-detected). → **Compute list** (reads the skip-memory; already-verified, unchanged bins are skipped). 3. Add team members (pre-filled from your user list). Leave **Levels** blank for full reach, or enter `1` / `1-2` for someone height-limited — their bins splice off to them. → **Auto-assign** → **Publish to team**. **Floor — each checker:** log in with their PIN → **My Checks** → **Load my list** → per card the **Apprise** line shows on-hand as `each | Case`, enter the **Count** in each and/or cases (or tick **MISSING**), then **Save** (green / red by demand), **Edit** to reopen, or **Blocked** (purple) if it can't be reached. **IC Manager — Dashboard:** - **Refresh totals** — cheap; per-member progress, good/short/missing/blocked. - **Load bin detail** (one read, reused by all four sub-tabs): - **Product totals** — located vs demand summed across every bin a product lives in; flags **COVERED** once enough is confirmed to fill the window's orders. - **Bin grid** — every bin, its Apprise qty, what was counted, status, who. - **Consolidation** — products spread across multiple bins, partial (loose) locations flagged, as a shortlist to combine. - **Outs / trace** — products still short of demand → launch bin traces. - **Apprise export (incorrect bins)** — Excel in your `Product Code / Location Prefix / Bin / / / UOM / Quantity / GenQtyAdj / Shorted /` format, one row per UOM, signed to correct Apprise to the physical count. **The Location Prefix is stamped per row from each card's division** (NH, EE, GT…), so a single export can cover several divisions at once. Example: a missing `23 each | 2 Case36` NH card exports `EA -23 / NH` and `Case36 -2 / NH`. Bins are written as text so they don't turn into dates. - **Save day to memory & archive** — folds today's counts into the skip-memory and closes the day. **Archive** — search any bin, code, or checker across all days. **Reports & KPIs** — pick any date range (or **Today / Last 7 days / Last 30 days**) and **Run** for a printable report: overall verified/good/short/missing/blocked plus eaches & cases counted, a **daily results** table (one row per day with a clean %), and **counter KPIs** per person (active days, bins checked, avg/day, good, outs, blocked, clean %, eaches, cases). **🖨 Print** opens a clean printable copy. This report is built from the small per-member stats records, so even a 30-day report costs only a few dozen reads. The report also lists every **off bin for ops** in the range (date, division, bin, code, name, demand, counted, status, who checked it) and has an **⬇ Off-bins to Excel** button to hand ops an actionable file. The off-bins list reads only the short/missing/blocked cards per day (one small query each), so it stays cheap. The first time you run it, Firestore will ask for a one-time index — open the link shown in the browser console, create it, and re-run. **Repairing inflated KPIs:** earlier counts could over-inflate because re-saving or editing a line to the *same* status double-counted that bucket (e.g. "96% good" alongside dozens "missing" that didn't add up). That counting is now fixed going forward. To correct days that were already recorded wrong, use **🔧 Rebuild stats** (pick the date, press it): it reads that day's cards once and recomputes every counter's totals from scratch, so the numbers reconcile again. ## Reads & writes (free-tier safe) The app never auto-polls, and now it **caches everything on the device** so the same data is never re-read: - **Checkers** load their list once. After that it's stored in the browser — navigating, going Back, saving, switching tabs, even closing and reopening the page cost **zero reads**. Only the **↻ Get latest** button does a fresh read, and you only need it if a manager changed your lines. - **The manager dashboard** caches **Load team entries** too; switching the Totals/Bins/Consolidation/Outs tabs and editing lines re-use the cached copy. **↻ Refresh** is the only thing that re-reads, so press it when you actually want the team's newest counts (use the cheap **Refresh totals** to watch progress in between). So across a day, reads ≈ each checker's single first load + a few manager refreshes — typically a few thousand, far under the 50k/day free limit. The header's **reads / writes** counters only move when something truly hits the server. ## Notes - **Indexes:** first time a checker loads or you load detail, Firestore may show a one-time link to create an index — click it (≈1 min), then retry. Normal Firestore setup, once per query type. - **Levels** are read from dash bins like 41-02-**02**. P-bins / bulk have no parseable level, so they go to unrestricted counters — tell me their level format if you want them splice-able. - **PINs** are stored in Firestore readable by the app; fine for an internal tool. For stronger control, enable Email/Password sign-in (commented option in `firestore.rules`). - **Divisions:** each division's data is kept separate (its own skip-memory, its own missing bins, its own Apprise prefix), so processing NH then EE the same day won't clobber each other. Run Build once per division; the dashboard and Apprise export span all divisions published that day. - **Demand window:** the app treats each upload as one window for the chosen division and measures coverage against the demand column you pick. If your demand export actually has separate columns per horizon (7/14/30…), point me at the headers and I'll switch coverage to read them directly.
